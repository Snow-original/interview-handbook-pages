# SKILL.md — 简历优化与面试突击 · 通用处理规范

> 本文件定义了一套可复用的简历优化方法论，适用于任何用户上传简历 + JD 的场景。
> 可作为 Agent System Prompt 的一部分，或作为独立 Skill 被调用。
> 版本：v1.0 | 来源：从黄熠伦简历优化工作中抽象泛化

---

## 一、这个 Skill 能做什么

| 任务 | 输入 | 输出 |
|------|------|------|
| 单岗位简历优化 | 原始简历 + 1份JD | 针对该JD优化后的 .docx 简历 |
| 多岗位简历拆分 | 原始简历 + N份JD | N份不同侧重的 .docx 简历 |
| 技能gap分析 | 简历 + JD | "还需要补的技能"清单 + 学习路径建议 |
| 面试突击手册 | JD + 简历 + 面试反馈（可选） | 带Q&A的 HTML 手册 |
| AGENT.md生成 | 对话历史 / 简历内容 | 结构化个人知识档案 |
| 面试反馈迭代 | 上一版手册 + 面试反馈 | 针对反馈强化的新版手册 |

## 二、核心处理原则

### 原则一：HR 视角优先
- 第一读者是不懂技术的 HR，第二读者才是技术面试官
- 项目描述必须回答："这个人能做什么"，而不是"这个人用了什么技术"
- 避免堆砌缩写和专业术语，优先用"解决了什么问题 + 结果是什么"结构

### 原则二：视角转换不改事实
- 同一个项目，针对不同岗位，改叙述角度，**不捏造经历**
- 数据分析岗：强调数据清洗、特征工程、结果可视化
- 算法岗：强调模型选型理由、消融实验、指标对比
- 产品岗：强调用户需求拆解、产品设计决策、数据驱动验证
- 开发岗：强调架构设计、技术选型、工程稳定性

### 原则三：荣誉与项目分离
- 荣誉奖项区：只写"竞赛名 + 奖项等级"，一行一条
- 竞赛的具体内容：转化为项目经历，用白话重新描述
- 规则：凡是竞赛，都要问"这个竞赛里你做了什么、解决了什么问题"

### 原则四：开发者视角（技术岗必须）
- 不只说"我用了XX"，要说"我为什么这样设计、有什么工程权衡"
- 每个技术决策都要能回答：为什么选这个方案、有什么代价
- 生产意识：成本控制、降级策略、可观测性、错误处理

### 原则五：主动补全技能 Gap
- 对比 JD 要求和用户简历，识别用户没有但 JD 要求的技能
- 分两类处理：
  - **可迁移**：用户有相关基础，用不同措辞描述即可（如用"数据管道"描述 Logstash 经验）
  - **真实缺口**：在"还需补充的技能"模块明确列出，给出学习优先级和路径

## 三、JD 解析流程

收到 JD 后，按以下维度提取信息：

```
1. 硬技能关键词（编程语言、框架、工具）
2. 软技能关键词（文档习惯、沟通、细心等）
3. 核心职责（最高频出现的动词：维护/分析/开发/协助）
4. 加分项（优先考虑/优秀者优先）
5. 隐含要求（从职责反推：写BOM维护 → 要求细心、Excel熟练）
```

**关键动作**：从 JD 的职责描述反推面试官真正在意什么，而不只是匹配关键词。

## 四、简历改写流程

### Step 1：读取并结构化原始内容
从用户简历中提取：
- 基本信息（姓名/院校/专业/GPA/联系方式）
- 教育经历（课程、荣誉）
- 所有项目经历（时间/角色/技术栈/工作内容/结果）
- 荣誉奖项（竞赛名/奖项）
- 技能列表

### Step 2：JD vs 简历 Gap 分析

```
对每一个 JD 要求，做三分类：
✅ 完全匹配：用户有直接经验，优先展示
⚠️ 部分匹配：用户有相关基础，需要角度转换
❌ 真实缺口：用户没有，列入"还需补充"清单
```

### Step 3：项目经历重写

每条项目描述使用以下模板：

```
【项目标题】  【时间】  【标签/岗位相关度标注】
技术栈：（按岗位相关度排序，最相关的放前面）
• [用白话描述做了什么] + [解决了什么问题] + [结果/数据]
• [对应JD哪条要求] + [体现了什么能力]
• （可选）[工程决策理由：为什么这样做而不是那样做]
```

**白话化改写规则：**
- 原文："基于 YOLOv8 的 Anchor-free 目标检测框架，引入 CBAM 注意力机制"
- 改写："针对小目标识别精度不足的问题，改进了图像检测模型（YOLOv8），加入注意力模块提升小目标的识别成功率，并通过对比实验验证了效果"

### Step 4：选择性填写原则

多岗位场景下，不同简历放不同项目：
- 每份简历项目数量：3-5 个，精不在多
- 选择标准：与 JD 关键词重叠度最高的项目优先
- 排列顺序：最匹配 JD 的项目放第一位

### Step 5：技能模块重组

按 JD 要求的优先级重新分类，例如：
- 数据分析岗：数据处理工具 → 数据库 → 编程语言 → 文档能力
- 算法岗：深度学习框架 → 机器学习算法 → 编程语言 → 数学基础
- Agent 产品岗：AI 工具使用 → 产品能力 → 开发语言 → 数据分析

## 五、技能 Gap 处理规范

### 呈现格式

在简历文档末尾，或单独输出一份 Gap 清单：

```
## 与 [岗位名称] JD 的技能差距分析

### ✅ 已具备（直接匹配）
- [技能] → 对应项目：[项目名]

### ⚠️ 可迁移（需要措辞转换）
- [JD 要求] → 你有的是 [相关经历] → 建议表述为 [改写方式]

### ❌ 真实缺口（建议补充）
优先级 HIGH：
- [技能名称]：[为什么重要] → 建议：[最快入门路径] → 预计时间：[X 天/周]

优先级 MEDIUM：
- [技能名称]：[简短说明]

优先级 LOW（有加分但不影响投递）：
- [技能名称]
```

### Gap 分析的诚实原则
- 不捏造用户没有的经历
- 对于真实缺口，给出"可以诚实说'正在学习中'"的措辞建议
- 区分"入门了解"和"工程实践"，不夸大

## 六、面试突击手册生成规范

### 手册结构

```
Header: 手册标题 + 面试反馈标注栏（如果有反馈）
Sidebar: 模块导航（核心模块置顶）
Content:
  - [如果有面试反馈] HOT 模块：直接针对反馈的专题 Q&A
  - 技术模块 1..N：每个模块 3-8 道题
  - 简历挂钩回答模板：把技术知识和用户项目串联的完整答法
Stats Bar: 总题数 / 已读题数 / 进度条
```

### Q&A 写作规范

每道题包含：
1. **问题**：模拟面试官的真实提问方式
2. **难度标注**：基础 / 重点 / 进阶 / 必答
3. **答案结构**：概念定义 → 工作原理 → 工程实践 → 简历关联
4. **简历关联框**（每道技术题必须有）：`🔑 简历关联：你在 XX 项目里做的 YY，就是这个原理的实践`
5. **开发者视角框**（技术岗适用）：`🛠 开发者视角：生产环境里这个问题的真实解法是...`
6. **代码示例**（需要时）：用深色背景代码块，语法高亮

### 手册分级

| 场景 | 模块数 | 题数 | 重点 |
|------|--------|------|------|
| 初次面试准备 | 5-7 | 30-40 | 基础概念 + 项目描述 |
| 面试反馈迭代 | +2 HOT 模块 | +10-15 | 直击反馈弱点 |
| 深度技术岗 | 8-10 | 50-65 | 工程细节 + 开发者视角 |

### 面试反馈迭代处理

收到面试反馈后的处理流程：

```
1. 识别反馈类型：
   - 概念不清楚 → 补充基础原理题
   - 实践经验不足 → 补充工程实现代码
   - 视角问题（应用者 vs 开发者）→ 补充"为什么这样设计"的题
   - 具体技术点 → 新增 HOT 专题模块

2. HR 视角反推：
   - 为什么面试官问这个问题？
   - 他们公司可能遇到了什么真实问题？
   - 他们期望候选人有什么能力？

3. 在手册顶部添加：
   - 面试反馈展示栏（黄色背景）
   - HR 视角分析（深色背景，3 卡片）
   - HOT 模块（红色标识）置顶
```

## 七、AGENT.md 生成规范

### 结构模板

```markdown
# AGENT.md — [用户姓名] · 个人知识档案 [vX.X]

## 一、基本信息（表格）
## 二、技术能力图谱（按方向分类）
  - 已有工程实践的技能（能用项目佐证的）
  - 掌握原理可在面试中表达的技能
  - 工具与环境
## 三、项目经历（完整版 · 开发者视角）
  每个项目包含：
  - 时间/角色/技术栈
  - 工程决策与设计要点（有编号的列表，不是散文）
  - 关键数据（指标/结果）
  - 如果升级/扩展怎么做（面试延伸回答素材）
## 四、荣誉与证书（表格）
## 五、面试经历与反馈记录（持续追加）
  - 每轮面试的反馈
  - HR 视角反推
  - 已针对优化的回答
## 六、目标岗位 JD 档案
## 七、写作 & 沟通偏好
## 八、Agent 使用说明（分场景）
  - 简历助手 Agent
  - 面试准备 Agent
  - 学习助手 Agent
  - 项目助手 Agent
```

### AGENT.md 的核心价值

1. **消除重复介绍**：新对话直接贴入，Agent 立即有完整上下文
2. **面试状态追踪**：记录面试了哪些公司、被问了什么、还差什么
3. **技能深度分级**：区分"工程实践"和"了解原理"，不夸大也不遗漏
4. **迭代成长记录**：每轮面试后追加反馈，档案越来越准确

## 八、文档输出规范

### 简历 .docx 生成

使用 `docx` npm 包（`npm install -g docx`），遵循以下设计规范：

**颜色方案（专业感）：**
```javascript
const BLUE = '2563EB';    // 主色，节标题背景
const DARK = '1E293B';    // 正文深色
const GRAY = '64748B';    // 辅助信息
const WHITE = 'FFFFFF';   // 标题文字
```

**页面设置：**
```javascript
// A4 页面
size: { width: 11906, height: 16838 }
margin: { top: 800, right: 1000, bottom: 800, left: 1000 }
```

**节标题样式（蓝色色块）：**
```javascript
new TextRun({
  text: section.title, bold: true, size: 26,
  color: WHITE, font: '微软雅黑',
  shading: { fill: BLUE, type: ShadingType.CLEAR }
})
```

**项目条目结构：**
```
项目标题（加粗，22pt）    时间（灰色）    标签（绿色色块）
技术栈：（斜体，灰色，19pt）
• 项目描述要点（20pt，正文色）
• 每条描述独立一个 bullet
```

**列表使用规范：**
```javascript
// 必须用 numbering config，禁止手动插入 "•" 字符
numbering: { config: [{ reference: 'bullets', levels: [{
  level: 0, format: LevelFormat.BULLET, text: '•',
  style: { paragraph: { indent: { left: 480, hanging: 240 } } }
}]}]}
```

### 面试手册 HTML 生成

**必要 UI 元素：**
- 固定侧边栏导航（含已读进度追踪）
- 统计卡片（总题数/已读/新增/模块数）
- 进度条（已读题数/总题数）
- 可折叠 Q&A 卡片（点击展开）
- 颜色编码：蓝色=普通题，红色=HOT/面试反馈题，绿色=新增题，紫色=简历模板题

**面试反馈栏（有反馈时必加）：**
```html
<!-- 深色背景，3列网格，显示每条反馈的反推分析 -->
<div class="insight-bar">
  <div class="insight-grid">
    <div>问：[反馈问题]  →  说明：[HR意图反推]</div>
    ...
  </div>
</div>
```

**代码块规范：**
- 深色背景 `#0F172A` / 文字 `#E2E8F0`
- 关键词蓝色 `#93C5FD` / 字符串绿色 `#86EFAC` / 注释灰色 `#64748B`
- 函数名黄色 `#FDE68A`
- `white-space: pre` + `overflow-x: auto`（防止代码折行）

## 九、完整工作流示例

### 场景：用户上传简历 + 3个JD + 后续面试反馈

```
Round 1 - 首次优化
├── 1. 读取简历，结构化提取所有项目
├── 2. 读取每个 JD，提取关键词和隐含要求
├── 3. 对每个 JD 做 Gap 分析（✅⚠️❌三分类）
├── 4. 生成 N 份简历（N=JD数量）
│   ├── 每份简历：选 3-5 个最匹配的项目
│   ├── 项目描述：针对该 JD 改写叙述角度
│   └── 技能模块：按 JD 优先级重新排序
├── 5. 输出 Gap 清单（每个岗位一份）
├── 6. 生成初版面试突击手册（HTML）
└── 7. 生成 AGENT.md（用于后续移植）

Round 2 - 面试反馈迭代
├── 1. 收到面试反馈（被问了哪些问题，哪里答得不好）
├── 2. HR 视角反推（为什么问这些？公司遇到了什么问题？）
├── 3. 识别弱点类型（概念/实践/视角/具体技术点）
├── 4. 新增 HOT 模块（直接针对反馈的专题）
├── 5. 更新手册（v2/v3...）
└── 6. 追加反馈到 AGENT.md 第五节
```

## 十、常见陷阱与规避方法

| 陷阱 | 错误做法 | 正确做法 |
|------|----------|----------|
| 捏造经历 | 用户没用过 RAG，但写"熟悉 RAG 开发" | 写"了解 RAG 原理，正在学习实践" |
| 过度技术化 | 堆砌缩写，HR 看不懂 | 先白话，再括号注技术名称 |
| 格式过度 | 每条都加粗，加 emoji | 只在必要时强调，保持简洁 |
| 项目太多 | 把所有项目都塞进简历 | 针对岗位精选 3-5 个 |
| Gap 否认 | 帮用户掩盖技能缺口 | 诚实列出，提供学习路径 |
| 脱离简历答题 | 技术题答完和用户经历没关系 | 每道技术题最后串联用户的具体项目 |
| 视角不足 | 只说"我用了 XX" | 说"我为什么用 XX，有什么权衡" |

## 十一、移植给其他用户的使用说明

使用本 Skill 时，按以下方式提供输入：

```
必须提供：
1. 用户简历（.docx / .pdf / 文字描述均可）
2. 目标岗位 JD（一个或多个）

可选提供（提升效果）：
3. 未写进简历的其他项目经历（文字描述或文档）
4. 面试反馈（被问了哪些问题、哪里没答好）
5. 用户偏好（希望强调哪些方向）

输出物：
- [岗位名称]_简历_[姓名].docx（每个JD一份）
- [岗位名称]_Gap分析_[姓名].md（可选）
- 面试突击手册_[姓名].html
- AGENT.md（个人知识档案）
```

**注意事项：**
- 本 Skill 不适合无项目经历的纯新手（需要有至少 1-2 个项目可以改写）
- 对于学术背景较强的用户，需要额外将论文/研究经历转化为工程语言
- 对于跨行业转岗用户，Gap 分析部分的优先级会更高
